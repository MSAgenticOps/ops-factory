# OpsFactory BI 智能分析与 Report Agent 特性能力规划

## 1. 背景与动机

### 1.1 业务痛点

企业 ITSM 运营产生大量工单数据（事件、变更、请求、问题），但数据消费链条存在三个断裂：

- **看不懂**：原始工单数据缺乏聚合与解读，SDM（服务交付经理）难以快速把握运营健康度
- **看不到**：固定仪表盘只能展示预设维度，无法按需钻取、交叉分析、自然语言追问
- **来不及**：月度/季度运营报告依赖人工整理 Excel，从数据提取到报告产出周期长达数天

### 1.2 为什么用 AI Native 方式解决

传统 BI 的交互模式是"人找数据"——用户需要理解数据结构、找到正确报表、手动筛选和解读。AI Native 的范式是"数据找人"——用户用自然语言表达业务问题，AI Agent 理解意图、选择分析路径、获取数据、生成洞察。

核心转变：

| 维度   | 传统 BI      | AI Native BI     |
| ---- | ---------- | ---------------- |
| 交互方式 | 点击仪表盘、拖拽筛选 | 自然语言对话           |
| 分析路径 | 预设报表、固定维度  | Agent 自主决策工具与参数  |
| 报告生成 | 人工撰写、模板填充  | Agent 基于数据自动生成   |
| 洞察深度 | 依赖分析师经验    | 内置风险雷达、异常检测、交叉关联 |
| 交付时效 | 天级         | 分钟级              |

### 1.3 规划目标

构建一个 **数据可计算、意图可理解、报告可交付** 的 AI Native ITSM 运营分析体系：

- 数据层：将四类 ITSM 工单原始数据结构化、可计算化
- 工具层：将分析能力封装为 AI Agent 可准确调用的意图驱动工具集
- 交互层：通过自然语言对话直接获取分析报告，零学习成本

## 3. 特性一：BI 智能分析引擎

### 3.1 定位

将 ITSM 工单原始数据转化为结构化、可计算、可查询的指标体系，同时服务人类仪表盘和 AI Agent 两类消费者。

### 3.2 为什么需要独立的分析引擎

AI Agent 的数据消费方式与传统仪表盘完全不同：

- **仪表盘**：一次性加载全量数据渲染 8 个 Tab，数据格式面向图表编码
- **AI Agent**：按意图精准获取特定域、特定维度的指标，数据格式面向理解和推理

如果只做仪表盘，Agent 每次都要加载全量数据再自行筛选——对 27B 参数的模型来说，上下文窗口和推理能力都会被浪费。独立的分析引擎让 Agent 像调 API 一样按需取数。

### 3.3 关键能力

#### C1: 八域聚合指标

覆盖 ITSM 运营的 8 个核心分析域，每个域输出一组预计算的 KPI：

| 域             | 核心指标                      | 业务意义           |
| ------------- | ------------------------- | -------------- |
| Executive     | 综合健康评分 (0-100)、等级、风险项     | 一句话回答"运营整体怎么样" |
| SLA           | 响应/解决达成率、违约数、P1/P2 达成率    | 衡量服务承诺兑现能力     |
| Incident      | 工单总量、P1/P2 占比、MTTR        | 衡量故障影响面和恢复速度   |
| Change        | 成功率、紧急变更占比、变更致事件率         | 衡量变更管控成熟度      |
| Request       | 完成率、CSAT、平均履行时长           | 衡量服务请求交付质量     |
| Problem       | 关闭率、RCA 率、已知错误数           | 衡量根因治理能力       |
| Cross-Process | 变更致事件率、脆弱度评分、48h P1/P2 窗口 | 衡量跨流程协同风险      |
| Workforce     | 人均吞吐、积压、SLA 达成、满意度        | 衡量团队能力和负载均衡    |

**业务效果**：SDM 不再需要逐个 Excel 表拉数据算指标，一个 API 调用获得全部 KPI。

#### C2: 多维度下钻

每个域支持多个分析维度的按需展开：

| 维度类型  | 示例                                           | 回答的问题               |
| ----- | -------------------------------------------- | ------------------- |
| 按优先级  | P1/P2/P3/P4 各自的 SLA 达成率                      | "高优先级的 SLA 为什么这么低？" |
| 按类别   | Database/Network/Application 各自的 MTTR        | "哪类故障修复最慢？"         |
| 按人员   | 每个 Resolver 的 SLA 达成率                        | "谁负责的工单违约最多？"       |
| 按时间   | 周度/月度的趋势变化                                   | "SLA 是在改善还是恶化？"     |
| 按风险等级 | High/Medium/Low 变更的成功率                       | "高风险变更的失败率是多少？"     |
| 按部门   | 各部门的请求满意度                                    | "哪个部门对服务最不满意？"      |
| 按根因   | Technical Defect / Process Gap / Human Error | "问题的根因集中在哪？"        |

**业务效果**：SDM 不满足于整体数字时，可以自然语言追问下钻——"按人员看呢？""按类别拆一下？"

#### C3: 时间序列趋势

支持按日/周/月粒度计算任意指标的时序数据，并支持多系列合并：

- SLA 趋势同时返回响应达成率 + 解决达成率 + P1/P2 达成率三条线
- Incident 趋势同时返回工单量 + SLA 达成率
- MTTR 趋势同时返回整体均值 + P1/P2 均值

**业务效果**：一眼看出趋势方向，而非孤立的单点数字。

#### C4: 跨流程关联追溯

ITSM 的四个流程不是孤立的。一次变更可能导致故障，一个故障可能暴露问题，一个问题可能关联多条请求。关联追溯能力包括：

- **变更 → 事件**：48 小时窗口内的 P1/P2 关联，识别"高危变更"
- **事件 → 问题**：重复发生的事件归类为已知错误
- **请求 → 事件**：请求激增预警，识别服务异常的早期信号
- **单据追溯**：输入工单号，自动关联上下游关联单据

**业务效果**：从"看单个工单"升级为"看关联链路"，识别系统性风险。

#### C5: 通用数据查询

对四类工单数据支持任意字段的过滤、排序、分页和聚合计算：

- 过滤支持等值、不等、包含、范围、IN 等操作符
- 聚合支持计数、均值、求和、百分比、分布
- 支持按任意字段 Group By

**业务效果**：当预计算指标无法满足个性化分析需求时，Agent 可以用通用查询自由组合。

### 3.4 验收标准

| #   | 验收项                    | 验证方法                                                                 |
| --- | ---------------------- | -------------------------------------------------------------------- |
| A1  | 8 个域的聚合指标可通过独立 API 获取  | 对每个域发送 GET /metrics/{domain}，返回 JSON 包含对应域的全部指标字段                    |
| A2  | 多维度下钻返回完整数据            | analyze_sla_rate(by_category=true, by_resolver=true) 返回按类别和按人员的完整达成率 |
| A3  | 趋势数据支持日期范围过滤           | 指定 startDate/endDate 后，返回的点数与时间范围匹配（如 2 个月约 9 个周点）                   |
| A4  | 跨流程关联可追溯               | 输入一个已知变更致事件的工单号，lineage 返回关联的事件工单                                    |
| A5  | 通用查询聚合结果正确             | compute_metric(metric=count, groupBy=Priority) 返回各优先级计数与原始数据一致       |
| A6  | SLA 达成率基于实际时间对比 SLA 标准 | 不依赖原始数据中的 SLA 标记字段，而是对比实际响应/解决时间与 SLA 标准                             |

## 4. 特性二：AI Native Report Agent

### 4.1 定位

一个理解自然语言分析意图、自主选择分析工具、生成数据驱动分析报告的 AI Agent。是 ITSM 运营数据与业务决策之间的智能桥梁。

### 4.2 为什么用 Agent 而不是模板化报告

| 维度   | 模板化报告     | AI Agent    |
| ---- | --------- | ----------- |
| 灵活性  | 固定格式、固定维度 | 按需分析任意维度组合  |
| 交互性  | 生成后不可追问   | 多轮对话、逐步下钻   |
| 洞察深度 | 数据罗列      | 基于风险雷达的主动洞察 |
| 个性化  | 所有人看同一份   | 针对问题生成针对性建议 |
| 门槛   | 需理解报告结构   | 零门槛，自然语言提问  |

### 4.3 关键能力

#### C1: 意图驱动的工具选择

Agent 的工具不是按数据域命名（如"获取 SLA 数据"），而是按分析意图命名（如"分析 SLA 达成率"）。工具名即分析概念，模型看到工具名就知道适用场景：

| 用户说什么         | Agent 选择什么工具                                    |
| ------------- | ----------------------------------------------- |
| "SLA 达成率怎么样？" | analyze_sla_rate                                |
| "最近故障工单量趋势？"  | analyze_incident_volume(by_time=true)           |
| "平均修复时间按人员看？" | analyze_mttr(by_resolver=true)                  |
| "变更成功率怎么样？"   | analyze_change_success_rate                     |
| "哪个部门满意度最低？"  | analyze_request_performance(by_department=true) |
| "问题根因分布？"     | analyze_problem_metrics(by_root_cause=true)     |
| "团队绩效排名？"     | analyze_workforce_performance                   |
| "给我月度概览"      | get_all_metrics                                 |

**业务效果**：用户不需要知道数据在哪个系统、用什么 API，只需用业务语言提问。

#### C2: 智能路由与参数默认值

Agent 调用工具时不需要指定所有参数。系统为每个分析场景预设合理默认值：

- 调 `analyze_sla_rate` 默认返回整体达成率 + 按优先级拆分
- 调 `analyze_workforce_performance` 默认返回 Top 10 人员
- 只有用户明确要求额外维度时，Agent 才需要设置对应参数

**业务效果**：即使模型参数量有限（27B），也能通过"默认够用、按需扩展"的设计保证基本交互质量。

#### C3: 多语言报告生成

Agent 自动匹配用户语言：

- 用户用中文提问 → 生成中文报告（标题、分析、建议均为中文）
- 用户用英文提问 → 生成英文报告
- 中途切换语言 → 后续响应跟随新语言

报告以 Markdown 格式保存为文件，同时在对话中输出摘要。报告内容必须：

- 所有数字和结论来自工具返回数据，禁止编造
- 包含分析期间和数据来源说明
- 聚焦关键发现（2-3 条洞察/节），不超过 3000 字
- 给出 3-5 条可操作建议

**业务效果**：SDM 用母语直接获得可分享的专业分析报告，从提问到交付分钟级完成。

#### C4: 数据驱动的风险洞察

Agent 内置风险雷达规则，在获取数据后自动识别风险信号：

| 风险信号 | 判定条件                      | 严重度       |
| ---- | ------------------------- | --------- |
| R001 | Incident SLA Rate < 70%   | CRITICAL  |
| R002 | P1/P2 MTTR > 24h          | CRITICAL  |
| R003 | Change Success Rate < 80% | WARNING   |
| R004 | 变更致事件率 > 10%              | WARNING   |
| R005 | Request CSAT < 3.0        | WARNING   |
| R006 | Problem 关闭率 < 50%         | ATTENTION |
| R007 | P1/P2 占比 > 20%            | ATTENTION |
| R008 | 紧急变更占比 > 15%              | ATTENTION |
| R009 | 问题已知错误无规避方案               | ATTENTION |
| R010 | 人员积压 > 20 单               | ATTENTION |

**业务效果**：Agent 不只是复述数据，而是主动识别风险、提出建议，从"报表工具"进化为"分析助手"。

#### C5: 工单详情与关联追溯

当 Agent 发现异常指标时，可以进一步下钻到具体工单：

- "给我 SLA 违约最严重的 5 条工单" → query_tickets
- "这条变更导致了哪些故障？" → trace_ticket_lineage
- "Database 类别的 MTTR 异常高，看下具体工单" → query_tickets(filters=[Category=Database])

**业务效果**：从指标到工单的闭环——发现异常 → 定位具体工单 → 追溯根因。

### 4.4 验收标准

| #   | 验收项           | 验证方法                                              |
| --- | ------------- | ------------------------------------------------- |
| B1  | 意图路由准确率 ≥ 90% | 测试 20 个典型自然语言问题，Agent 选择正确工具的比例                   |
| B2  | 单指标分析一次调用完成   | "SLA 达成率怎么样" → Agent 仅调用一次 analyze_sla_rate 即完成分析 |
| B3  | 报告内容数据准确      | 报告中所有数字与 BI API 直接返回的原始数据一致                       |
| B4  | 报告语言匹配用户语言    | 中文提问 → 全中文报告；英文提问 → 全英文报告                         |
| B5  | 报告包含分析期间      | 报告明确标注数据覆盖的起止时间（来自 dataDateRange）                 |
| B6  | 追问可下钻         | 第一轮"SLA 怎么样"，追问"按人员看呢"→ Agent 调用 by_resolver 维度   |
| B7  | 非相关请求拒绝       | 问"帮我订会议室"→ Agent 礼貌拒绝并说明只支持 ITSM 运营分析             |
| B8  | 报告文件可下载       | 报告保存为 .md 文件，通过对话中的链接可直接访问                        |

## 5. 联合验收场景

以下端到端场景验证 BI 引擎 + Report Agent 的协同效果：

### 场景 1: 月度运营概览

```
用户: "给我上个月的运维运营概览"

预期 Agent 行为:
1. 调用 get_all_metrics(startDate=上月1号, endDate=上月末)
2. 获取 dataDateRange 确认分析期间
3. 从返回数据提取各域关键指标
4. 生成报告，包含: 健康评分、KPI 总览表、Top 3 风险、建议
5. 保存报告文件，输出摘要

验收:
- 报告覆盖全部 4 个流程域
- 所有数字可溯源到 API 返回数据
- 报告 < 3000 字
```

### 场景 2: 异常下钻分析

```
用户: "为什么上个月 SLA 这么低？"
追问: "哪些人员违约最多？"
追问: "给我违约最多的那个人的具体工单"

预期 Agent 行为:
1. analyze_sla_rate(by_priority=true) → 发现 P1/P2 解决达成率极低
2. 追问时: analyze_sla_rate(by_resolver=true) → 找到 SLA 达成率最低的人员
3. 再次追问: query_tickets(filters=[Resolver=该人员, SLA=违约], sortBy=ResolutionTime desc)

验收:
- 每轮追问正确选择工具和参数
- 最终定位到具体工单记录
- 全程中文
```

### 场景 3: 跨流程关联分析

```
用户: "最近有变更导致故障吗？"

预期 Agent 行为:
1. analyze_change_success_rate(by_time=true) → 获取变更致事件趋势
2. 如果发现异常: query_tickets(filters=[Incident Caused=Yes]) → 获取具体变更
3. 对关键变更: trace_ticket_lineage → 追踪关联的事件链

验收:
- 报告包含变更致事件率和趋势
- 给出具体变更工单和对应故障工单
```

## 6. 非功能要求

| 维度        | 目标                                     |
| --------- | -------------------------------------- |
| 报告生成时效    | 单次分析报告 < 60 秒（含工具调用 + LLM 生成）          |
| BI API 响应 | 单域指标 < 2s，全量指标 < 10s                   |
| 数据刷新      | 仪表盘 Snapshot 缓存，支持手动强制刷新               |
| 工具选择准确率   | 20 个标准场景 ≥ 90%（基于 qwen3.5-27B）         |
| 报告准确性     | 所有数字和结论 100% 来自工具数据，零编造                |
| 语言支持      | 中文、英文，自动匹配                             |
| 模型兼容性     | 工具设计对 27B+ 参数模型友好（意图映射 + 默认参数 + 短工具描述） |

## 7. 技术约束

| 约束      | 说明                                              |
| ------- | ----------------------------------------------- |
| 数据源     | 当前为 Excel 文件加载，非实时数据库                           |
| 模型      | Report Agent 使用 qwen3.5-27B，工具设计需适配中等参数量模型的能力边界 |
| 单次交互上下文 | Agent 上下文窗口 65K tokens，max output 32K tokens    |
| MCP 超时  | 单次工具调用超时 60 秒                                   |
| 数据量     | 约 9000+ incidents、50 changes/requests/problems  |

## 8. 演进路线

### Phase 1: 当前已完成 — 可计算底座 + Agent 基础交互

- BI 分析引擎 8 域聚合指标 + 多维度下钻 + 趋势 + 查询 + 关联
- Report Agent 11 个意图驱动工具 + 智能路由 + 中文/英文报告生成
- 仪表盘前端 8 Tab 可视化

### Phase 2: 数据实时化 + 分析深度化

| 能力                           | 业务价值                      |
| ---------------------------- | ------------------------- |
| 接入 ITSM 系统实时数据（API 替代 Excel） | 消除数据延迟，实时运营感知             |
| 同比/环比自动计算                    | "SLA 比上月改善了还是恶化了？"        |
| 指标阈值告警                       | SLA 低于目标值时 Agent 主动推送风险提醒 |
| 多租户数据隔离                      | 不同客户/项目独立分析               |

### Phase 3: 主动智能 + 知识积累
