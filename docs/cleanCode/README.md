# 整洁代码审查规则

本目录存放从公司整洁代码电子表格文件中提取的代码审查规则。每个来源电子表格文件对应一份规则文档。

这些文档用于合入请求阶段的人工智能代码审查，不建议作为编码阶段的常驻上下文。审查时根据变更语言或构建面选择对应规则集，并与内置代码审查结果合并。

## 合入请求审查使用方式

1. 先运行常规代码审查，检查正确性、回归风险和测试覆盖。
2. 根据变更文件类型，从本目录加载对应的整洁代码规则集。
3. 只检查适用于变更代码或判断变更所必需的相邻代码的规则。
4. 在合入请求描述中列出已检查的规则集，并按来源规则名记录发现的问题。
5. `提示`、`信息` 和建议类规则也按公司要求纳入整改；若确实无法修改，需要在合入请求描述中说明例外原因。

## 使用约束

这些规则作为审查参考整体是低风险的，但不能脱离上下文机械套用。审查时必须满足以下约束：

1. 只对本次变更涉及的语言、构建配置或安全场景加载对应规则。
2. 只报告有明确代码证据的问题，不根据规则名称做推断。
3. `通用规范建议类`、`代码坏味道类`、`信息` 和 `提示` 也需要应改尽改；审查输出中可按较低风险标注，但不能直接忽略。
4. 构建选项规则只适用于构建脚本、编译参数或交付构建配置变更。
5. 与当前仓库既有编码风格冲突的格式类规则，以仓库现有约定为准。
6. 安全类规则优先级高，但仍需要结合数据来源、信任边界和运行环境判断。
