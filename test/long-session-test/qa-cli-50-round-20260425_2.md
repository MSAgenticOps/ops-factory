## Compaction 纠正

原报告中的 `Compaction rounds: 1, 2, ..., 17` 是错误统计，不表示每轮都发生了压缩。

准确结论：

- 明确发生 compaction 的轮次：第 `7` 轮
- 第 7 轮 SSE 事件中出现：`UpdateConversation: 1`
- goosed 日志中出现：`Context Compaction`
- goosed 日志中出现：`Your context was compacted...`
- 第 8 轮以后，单轮 LLM 请求输入 token 明显下降，符合第 7 轮压缩后继续运行的现象

错误原因：

- 旧报告把 `logCompactionSignals` 也计入了 `Compaction rounds`
- `logCompactionSignals` 的匹配条件过宽，把 `Session tokens`、`input_tokens`、`truncated`、`save tokens`、tool-pair summarization 里的普通 token 文本也识别成了压缩信号
- 旧脚本扫描日志片段时没有严格按“本轮新增的真实 compaction 事件”过滤，导致第 7 轮之后的同一条 compaction 日志被重复归入后续轮次

后续判断 compaction 应以这些信号为准：

1. SSE `eventCounts.UpdateConversation > 0`
2. goosed 日志明确包含 `Context Compaction`
3. goosed 日志明确包含 `Your context was compacted`

## 第 17 轮失败分析

第 17 轮不是 submit 失败，也不是 SSE 一开始就断开：

- Submit 耗时：`27 ms`
- 首个 SSE 事件耗时：`2,082 ms`
- 首个内容耗时：`2,082 ms`
- 已收到 `Message:11`
- 已调用工具：`knowledge-cli__read_file` 1 次
- 未收到 terminal 事件：没有 `Finish`，也没有 `Error`

结合 MCP 日志，`read_file` 工具调用已经快速完成。问题更可能发生在工具返回后的 LLM/provider 继续生成阶段，或 tool-pair summarization / goosed 后续处理阶段。

## 观察到的风险点

1. `GOOSE_TOOL_PAIR_SUMMARIZATION` 相关请求较多。
2. 部分 tool-pair summarization 日志中出现较长的 thinking / token saving 推理内容。
3. 这类摘要本意是节省上下文，但如果摘要请求自身输出很长，会反过来增加延迟和 token 消耗。
4. 第 17 轮超时发生在工具返回之后，符合“post-tool 生成或摘要链路卡住”的风险形态。

## 建议

1. 收紧 `GOOSE_TOOL_PAIR_SUMMARIZATION` 提示词：只允许输出短摘要，禁止输出推理过程，限制最大长度。
2. 对小型、已截断且结构化的 knowledge-cli 工具返回跳过 tool-pair summarization，避免不必要的二次 LLM 调用。
3. 增加 post-tool stall 检测：工具返回后如果 N 秒内没有 assistant token 或 terminal event，应记录结构化错误，必要时 cancel/retry。
4. 后续报告统计中应把以下指标分开：
   - 当前请求 token
   - session 累计 token
   - compaction 事件
   - tool-pair summarization 事件
   - 工具调用耗时
   - LLM/provider 耗时
