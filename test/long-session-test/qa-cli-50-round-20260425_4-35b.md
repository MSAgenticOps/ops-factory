## 观察到的风险点

1. QA CLI Agent 的主要压力来自知识库搜索/读文件工具调用后的模型继续生成阶段。
2. 如果出现超时且工具已快速返回，需要重点排查 provider 续写、tool-pair summarization、goosed 事件流终止信号。
3. `GOOSE_MAX_TURNS=50` 与连续单 session 50 轮接近边界，后半段需要关注 compaction 是否稳定触发。
4. reasoning/thinking 统计只记录事件流里的结构化 content block，不包含普通文本。

## 建议

1. 保留 per-tool 延迟、post-tool 生成耗时和 compaction 事件三类指标，避免把模型慢误判为工具慢。
2. 若后续继续比较 27b/35b，建议固定同一 query、同一 session 策略和同一单轮超时。
3. 对超时轮次进一步查对应 goosed 日志和 `llm_request*.jsonl`，确认卡在 provider 还是 goosed 后处理。
